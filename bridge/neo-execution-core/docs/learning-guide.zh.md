# neo-execution-core 技术学习指南

这份指南把 `neo-execution-core` 当作 Neo N4 的一个技术单元来解释。它不是源码阅读图，而是帮助读者理解：这个单元负责什么、哪些技术假设保证它正确、数据如何移动、状态如何变化、证据如何被验证、它如何接入 Neo N4 的整体架构。

## 技术契约

| 维度 | 含义 |
| --- | --- |
| 层级 | N4 批处理执行核心 |
| 目的 | 后端无关的 L2 批处理状态转换核心，供快速执行和证明生成复用。 |
| 输入 | L2 批次 <br> 前序状态根 <br> 执行参数 |
| 职责 | 校验批次结构 <br> 执行确定性状态转换 <br> 提交新状态根 |
| 输出 | 执行轨迹 <br> 新状态根 <br> 公开证明输入 |
| 消费方 | neo-zkvm-guest <br> neo-zkvm-host <br> 网关服务 |

## 架构模型

`neo-execution-core` 接收 L2 批次 | 前序状态根 | 执行参数，拥有的边界是：校验批次结构 | 执行确定性状态转换 | 提交新状态根。它输出 执行轨迹 | 新状态根 | 公开证明输入，然后由 neo-zkvm-guest | neo-zkvm-host | 网关服务 消费。

分层规则：每层只承担一个清晰技术责任。

## 工作流

1. 接收批次
2. 规范化输入
3. 运行状态转换
4. 哈希输出
5. 发布证据

失败路径：输入不满足约束或输出无法验证。

## 数据流

1. 交易
2. 执行核心
3. 轨迹 + 承诺
4. 证明/公开输出

承诺信号：输入摘要、转换摘要和输出摘要。

## 状态、证明和信任

- 状态转换：显式规则把已接受输入转换成可检查输出。
- 终局条件：消费方接受输出。
- 信任模型：只信任通过验证边界的数据。
- 验证边界：所有输入进入显式校验边界。
- 重放与顺序：输入摘要和输出摘要绑定同一上下文。

## 集成和运行

- NeoFS DA：NeoFS 保存批次数据、见证或轨迹摘要以及可取回证据。
- 证明系统：证明系统把 L2 执行声明压缩为可验证证据。
- Gateway/API：Gateway 负责用户路由、查询、提交和健康状态聚合。
- 桥与异构链：桥规则统一 L1-L2、L2-L2 和异构链消息与资产。
- 可观测证据：输入摘要、输出摘要、状态变化和拒绝原因。
