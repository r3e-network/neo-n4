# neo-n4-sdk 技术学习指南

这份指南把 `neo-n4-sdk` 当作 Neo N4 的一个技术单元来解释。它不是源码阅读图，而是帮助读者理解：这个单元负责什么、哪些技术假设保证它正确、数据如何移动、状态如何变化、证据如何被验证、它如何接入 Neo N4 的整体架构。

## 技术契约

| 维度 | 含义 |
| --- | --- |
| 层级 | 开发者 SDK |
| 目的 | 用于构建访问 Neo N4 API 的工具和服务的 Rust SDK。 |
| 输入 | 开发者应用 <br> 网关端点 <br> 钱包/配置 |

| 输出 | 强类型客户端结果 <br> 交易请求 <br> 查询响应 |
| 消费方 | 应用 <br> 运维工具 <br> 集成测试 |

## 架构模型

`neo-n4-sdk` 接收 开发者应用 | 网关端点 | 钱包/配置，拥有的边界是：编码 API 请求 | 处理桥/证明模型 | 返回强类型结果。它输出 强类型客户端结果 | 交易请求 | 查询响应，然后由 应用 | 运维工具 | 集成测试 消费。

分层规则：界面、编码、网络提交和诊断分层。

## 工作流

1. 创建客户端
2. 构造请求
3. 签名或查询
4. 提交网关
5. 解码响应

失败路径：配置无效、签名策略失败、网络拒绝或输出校验失败。

## 数据流

1. 应用意图
2. SDK 模型
3. RPC payload
4. Neo N4 服务响应

承诺信号：请求摘要、网络 profile 和输出产物哈希。

## 状态、证明和信任

- 状态转换：请求、配置、包或报告通过校验阶段推进。
- 终局条件：目标服务接受请求并返回可追踪结果。
- 信任模型：信任校验和签名策略，不信任未校验用户输入。
- 验证边界：配置、参数、签名、网络和输出格式通过校验。
- 重放与顺序：请求 id、网络 id 和签名域避免重复或跨网误用。

## 集成和运行

- NeoFS DA：NeoFS 保存批次数据、见证或轨迹摘要以及可取回证据。
- 证明系统：证明系统把 L2 执行声明压缩为可验证证据。
- Gateway/API：Gateway 负责用户路由、查询、提交和健康状态聚合。
- 桥与异构链：桥规则统一 L1-L2、L2-L2 和异构链消息与资产。
- 可观测证据：命令、配置摘要、网络、输出产物和诊断。
